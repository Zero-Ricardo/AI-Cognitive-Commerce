# AI 商品采购平台需求文档

## 1. 文档说明

- 产品定位：在传统电商交易前链路上逐步增加 AI 导购能力的商品采购平台。
- 当前交付范围：第一阶段传统电商 MVP。
- 第二阶段增量：Elasticsearch 商品检索、Redis 热榜与缓存、历史浏览和热销指标体系；AI 导购仅制作前端 Mock 原型。
- 后续阶段：AI 自然语言搜索、推荐解释与独立 Python AI 服务。

## 2. 产品目标

第一阶段先建立稳定、可维护的商品数据和用户行为闭环，让用户能够注册登录、查找商品、收藏和管理购物车，让管理员能够维护商品、分类与品牌。商品的关键词、场景、适用人群和规格采用结构化字段保存，为第二阶段 AI 检索提供可靠数据基础。

## 3. 用户角色

### 3.1 游客

- 浏览首页、商品列表和商品详情。
- 使用关键词、分类、品牌、价格、库存及排序条件搜索商品。
- 注册或登录账号。

### 3.2 注册用户

- 具备游客全部能力。
- 收藏或取消收藏商品。
- 添加、修改、选择和删除购物车商品。
- 查看个人资料并退出登录。

### 3.3 管理员

- 登录独立管理后台。
- 新增、编辑、查询、上下架和逻辑删除商品。
- 维护商品分类和品牌。
- 查看商品与用户行为概览。

## 4. 第一阶段功能范围

### 4.1 用户与认证

- 用户可通过用户名、手机号或邮箱注册，用户名必须唯一。
- 密码需进行不可逆加密，不得明文保存或返回。
- 登录成功后使用 HttpOnly Cookie 维持访问令牌和刷新令牌。
- 普通用户和管理员权限隔离；管理接口只允许管理员访问。
- 用户可查询当前登录信息并主动退出。

### 4.2 商品浏览

- 首页展示导航、类目入口和热销商品。
- 商品卡片展示主图、名称、副标题、分类、品牌、价格和库存状态。
- 商品详情展示多图、描述、规格、适用场景、适用人群及库存。
- 仅已上架且未删除的商品可在 C 端访问。

### 4.3 传统搜索与筛选

- 支持按商品名称、商品编号、关键词和副标题模糊搜索。
- 支持分类、品牌、最低价、最高价和是否有货筛选。
- 支持综合、价格升序、价格降序、最新和热度排序。
- 支持分页并展示结果总数。
- 示例“想买苹果，预算 30 元以内”在第一阶段可通过关键词“苹果”与最高价 30 的组合筛选完成。

### 4.4 收藏

- 登录用户可收藏或取消收藏商品。
- 同一用户不能重复收藏同一商品。
- 收藏列表支持分页并展示商品当前状态。

### 4.5 购物车

- 登录用户可将已上架商品加入购物车。
- 同一商品重复加入时累加数量，但不得超过实时库存。
- 支持修改数量、单项选择、全选和删除。
- 购物车实时计算商品数量、选中数量和选中商品总价。
- 商品下架或库存不足时需要明确标记为不可用。

### 4.6 商品管理后台

- 商品字段包括编号、名称、副标题、分类、品牌、售价、原价、库存、主图、多图、描述、关键词、场景、人群、规格和状态。
- 商品编号唯一；价格与库存不得为负数；上架商品必须具备必要展示信息。
- 支持草稿、已上架、已下架三个状态。
- 编辑和状态变更使用版本号防止并发覆盖。
- 删除采用逻辑删除并记录操作审计。

### 4.7 分类与品牌管理

- 分类支持两级结构、排序和启停状态。
- 品牌支持名称、Logo、说明、排序和启停状态。
- 已被有效商品引用的分类或品牌不可直接删除。

## 5. 业务规则

- 金额统一以十进制定点类型保存与计算，前端只负责展示。
- 时间由服务端生成并按统一时区持久化。
- 热销榜第一阶段基于浏览、收藏、加购等行为加权计算；数据不足时由最新上架商品补位。
- 商品图片第一阶段支持 URL 与本地文件存储，生产环境可替换为对象存储。
- 所有接口使用统一响应结构，并返回可追踪的请求编号。

## 6. 非功能需求

- 安全：密码使用 BCrypt；认证 Cookie 为 HttpOnly；接口进行权限、参数和文件类型校验。
- 性能：商品列表必须分页；常用查询字段建立索引；首页限制热销商品数量。
- 可维护性：数据库结构由 Flyway 管理，前后端共享接口类型，模块按业务域组织。
- 可部署性：MySQL、后端、C 端和管理端均可通过 Docker Compose 一键启动。
- 可演进性：第一阶段不引入 PostgreSQL、Redis、Elasticsearch 和 Python 微服务，但保留热榜接口及 AI 所需商品标签字段。

## 7. 第二阶段规划（搜索与数据基础设施）

- Elasticsearch 承担商品关键词检索、属性过滤、聚合、联想、高亮和相关性排序。
- 商品数据仍以 MySQL 为事实来源，通过 Outbox 增量同步到 Elasticsearch。
- Redis 承担全局及分类热销榜、热点商品摘要和短期缓存；Redis 数据必须可从 MySQL 重建。
- 增加登录用户历史浏览列表，支持查询、单条删除和清空；游客历史默认保存在浏览器本地。
- 热销商品由销量、有效浏览量和当前有效收藏量共同决定，初始权重分别为 50%、30% 和 20%。
- 当前没有订单模块时，销量只允许来自明确标识的初始化数据或可审计的外部导入，不得以加购量代替销量。
- Elasticsearch 或 Redis 不可用时降级到 MySQL 基础搜索和热销快照，不影响第一阶段商城核心功能。
- 前端可以实现 AI 导购 Mock 原型，但必须标识为演示数据，不调用真实 AI 后端。

## 8. 后续 AI 阶段规划

- 用户以自然语言表达模糊需求，例如“想买旅行用相机，预算 500 以内”。
- AI 服务提取品类、预算、场景、人群和偏好等检索条件。
- Elasticsearch 召回候选商品，AI 对候选商品进行重排并生成推荐理由。
- AI 后端、会话存储、模型调用和流式响应均不属于第二阶段。

## 9. 第一阶段验收标准

- 游客能够浏览覆盖 10 个一级类目的 30 条初始化商品，并可使用关键词、分类和价格组合筛选。
- 初始化数据包含 15 个测试品牌、5 个测试用户、8 条虚构收货地址及 30 张可追溯授权来源的本地实拍图。
- 普通用户能够登录、收藏商品并完成购物车增删改选。
- 管理员能够登录后台并维护商品、分类和品牌。
- 非管理员访问管理接口得到拒绝响应。
- 前后端生产构建、后端测试和 Docker 镜像构建全部成功。
- 新环境执行 `docker compose up --build` 后可自动创建表结构和演示数据。
