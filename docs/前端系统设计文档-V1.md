# AI 商品采购平台第一阶段前端系统设计文档

> 文档版本：V1.0  
> 阶段范围：第一阶段——传统电商 MVP  
> 技术基线：React + TypeScript + Vite + Ant Design  
> 原型参考：`AI商品采购原型图/`

## 1. 文档目的

本文档说明第一阶段前端系统的工程结构、页面组成、组件划分、状态管理、接口协作、权限控制、异常处理和测试方案，作为前端开发、联调和验收依据。

第一阶段交付一个不依赖 AI、Redis 和 Elasticsearch 也能完整运行的传统电商平台。AI 搜索相关原型仅作为后续扩展参考，本阶段不提供可用的 AI 搜索入口。

## 2. 建设范围

### 2.1 C 端商城

- 用户注册、登录、退出和登录状态恢复。
- 首页商品展示。
- 商品分类、关键词搜索、筛选、排序和分页。
- 商品详情。
- 商品收藏与取消收藏。
- 购物车增、删、改、查及金额计算。
- 个人资料与收藏列表。

### 2.2 管理后台

- 管理员登录和后台访问控制。
- 商品列表、查询、新增、编辑、查看、上下架和逻辑删除。
- 分类管理。
- 品牌管理。
- 商品图片、标签、使用场景、适用人群和规格参数维护。

### 2.3 本阶段不实现

- AI 自然语言搜索、AI 对话和推荐理由。
- Elasticsearch 搜索。
- Redis 热门榜和缓存。
- 下单、支付、物流、退款和售后。
- 优惠券、促销、评价和商家入驻。

## 3. 技术选型

| 类别 | 选型 | 用途 |
|---|---|---|
| UI 框架 | React + TypeScript | 页面及组件开发 |
| 构建工具 | Vite | 本地开发和生产构建 |
| 组件库 | Ant Design | 表格、表单、分页、弹窗等 |
| 路由 | React Router | C 端及后台路由管理 |
| 服务端状态 | TanStack Query | 请求缓存、失效刷新、加载状态 |
| 客户端状态 | Zustand | 登录用户和少量全局 UI 状态 |
| HTTP 客户端 | Axios | API 请求、拦截器和错误转换 |
| 样式 | Ant Design Token + CSS Modules | 主题统一和局部样式隔离 |
| 包管理 | pnpm workspace | 多应用和共享包管理 |
| 单元测试 | Vitest + React Testing Library | 组件和业务逻辑测试 |
| 端到端测试 | Playwright | 核心用户流程测试 |
| 代码质量 | ESLint + Prettier | 静态检查和格式统一 |

依赖版本在项目初始化时锁定到 `pnpm-lock.yaml`，禁止生产构建使用浮动的 `latest` 版本。

## 4. 总体架构

前端采用一个仓库、两个应用、多个共享包的 Monorepo 结构：

```text
frontend/
├── apps/
│   ├── mall-web/                 # C端商城
│   │   └── src/
│   │       ├── app/
│   │       ├── pages/
│   │       ├── features/
│   │       ├── layouts/
│   │       └── main.tsx
│   └── admin-web/                # 管理后台
│       └── src/
│           ├── app/
│           ├── pages/
│           ├── features/
│           ├── layouts/
│           └── main.tsx
├── packages/
│   ├── api-client/               # 请求封装、DTO类型、错误模型
│   ├── shared-components/        # 公共展示组件
│   ├── design-tokens/            # 主题变量
│   ├── shared-types/             # 跨应用类型
│   └── utils/                    # 通用工具
├── package.json
├── pnpm-workspace.yaml
└── tsconfig.base.json
```

### 4.1 分层约束

```text
Page 页面编排
  ↓
Feature 业务模块
  ↓
Query/Mutation Hook
  ↓
API Client
  ↓
Spring Boot REST API
```

- 页面不直接调用 Axios。
- 组件不直接拼接接口地址。
- 服务端数据统一通过 TanStack Query 管理。
- 金额展示由公共格式化函数处理，不能使用浮点数自行累计作为最终结算依据。
- 后端返回的 DTO 与页面展示模型可以通过 Adapter 转换，避免页面绑定数据库字段。

## 6. 路由设计

### 6.1 C 端路由

| 路由 | 页面 | 访问要求 |
|---|---|---|
| `/` | 商城首页 | 公开 |
| `/login` | 登录 | 公开 |
| `/register` | 注册 | 公开 |
| `/products` | 商品列表/搜索结果 | 公开 |
| `/products/:productId` | 商品详情 | 公开 |

商品列表的条件必须可序列化到 URL，例如：

```text

```

刷新、复制链接和浏览器前进后退时应恢复相同查询结果。

| 路由 | 页面 | 权限 |
|---|---|---|
| `/admin/login` | 管理员登录 | 公开 |
| `/admin` | 后台概览 | ADMIN |
| `/admin/products` | 商品管理 | ADMIN |
| `/admin/products/new` | 新增商品 | ADMIN |

## 7. C 端页面设计

### 7.1 公共布局

`MallLayout` 包含：

- Logo 和首页入口。
- 商品搜索框。
- 分类导航。
- 收藏、购物车和用户菜单。
- 购物车数量徽标。
- 主内容区域。
- 页脚。

移动端将导航收起，搜索框保持易访问，商品网格由桌面四列降为平板两列、手机一至两列。

### 7.2 登录与注册

登录表单字段：账号、密码。注册字段：用户名、手机号或邮箱、密码、确认密码。

交互要求：

- 使用 Ant Design Form 完成必填、格式和长度校验。
- 密码不写入日志、URL 和持久化状态。
- 登录成功后刷新当前用户数据，并跳转原目标页或首页。
- 401 响应触发登录态清理；涉及未保存内容时先提示。
- 重复注册、密码错误等使用后端业务错误消息，不暴露内部异常。

### 7.3 首页

页面模块：

- 顶部搜索。
- 商品分类快捷入口。
- 推荐/精选商品区。
- 新上架商品区。
- 热门商品区。

第一阶段的“热门商品”由后端根据浏览、收藏和加购统计返回；若尚未积累数据，则按后台配置或上架时间降级展示。

### 7.4 商品列表与传统搜索

筛选条件：

- 关键词。
- 分类。
- 品牌。
- 最低价、最高价。
- 是否有库存。

排序方式：综合、价格升序、价格降序、最新上架、热度。

交互规则：

- 条件变化后页码重置为 1。
- 桌面使用侧边筛选栏，移动端使用抽屉。
- 已应用条件展示为可关闭标签。
- 搜索框回车和搜索按钮行为一致。
- 请求期间保留已有结果并展示局部加载状态，避免页面跳动。
- 无结果时展示清空筛选和返回分类入口。

`ProductCard` 展示主图、名称、简述、价格、品牌、库存状态、收藏和加购操作。本阶段不展示 AI 匹配度或 AI 推荐理由。

### 7.5 商品详情

页面组成：

- 商品图片预览。
- 名称、副标题、价格和库存。
- 品牌、分类和标签。
- 规格参数。
- 商品描述。
- 数量选择。
- 收藏和加入购物车。

操作前端校验：数量必须为正整数且不超过页面所知库存；后端执行最终库存校验。商品下架时禁用加购并提示状态。

### 7.6 购物车

功能：

- 查看购物车条目。
- 增加或减少数量。
- 删除条目。
- 单选、全选。
- 显示选中商品数量和总金额。
- 识别下架、库存不足和价格变化。

更新数量采用乐观交互需谨慎：请求失败必须回滚。总金额以服务端返回结果为最终依据。第一阶段不实现结算，页面明确显示“订单功能将在后续阶段开放”，避免出现无后续动作的可用结算按钮。

### 7.7 收藏和个人中心

收藏列表支持分页、取消收藏、查看详情和加入购物车。个人中心第一阶段只展示和修改基础资料，不展示地址、支付方式和订单等未实现模块。

## 8. 管理后台页面设计

### 8.1 后台布局

`AdminLayout` 包含左侧菜单、顶部用户区、面包屑和内容区域。菜单根据权限生成，当前阶段只有 `ADMIN` 角色。

### 8.2 商品列表

查询项：商品名称、商品编号、分类、品牌、状态和价格区间。

表格字段：商品信息、分类、品牌、价格、库存、状态、更新时间和操作。

操作：查看、编辑、上架、下架、删除。危险操作使用二次确认；上下架和删除成功后使列表查询缓存失效。

### 8.3 商品编辑表单

按区域组织：

1. 基础信息：名称、副标题、分类、品牌。
2. 销售信息：销售价、原价和库存。
3. 媒体信息：主图和图片集。
4. 搜索信息：关键词、标签、场景和适用人群。
5. 详细信息：描述和规格参数。
6. 状态信息：草稿、上架或下架。

规格参数使用可增删的键值列表，提交时转换为后端约定的 JSON 结构。价格控件只允许两位小数，库存只允许非负整数。

图片上传采用先上传后绑定：上传接口返回文件标识和访问地址，保存商品时提交文件标识。编辑未保存就离开时提示用户。

### 8.4 分类与品牌

- 分类支持最多两级树形结构、排序和启停。
- 存在子分类或关联商品时禁止直接删除。
- 品牌支持名称、Logo、说明、排序和启停。
- 禁用分类或品牌不影响已有历史关联，但新增/编辑商品时不可再选择。

## 10. 状态与数据流

### 10.1 服务端状态

推荐 Query Key：

```ts
['current-user']
['products', searchCondition]
['product', productId]
['categories']
['brands']
['cart']
['favorites', pageCondition]
['admin-products', searchCondition]
```

Mutation 后的刷新规则：

- 收藏变化：刷新商品详情、相关商品列表和收藏列表。
- 购物车变化：刷新购物车和顶部购物车数量。
- 商品编辑：刷新商品详情、商品列表及后台列表。
- 商品上下架：刷新后台列表和相关 C 端商品缓存。

### 10.2 客户端状态

Zustand 只保存：

- 当前用户的轻量视图。
- 登录状态初始化标志。
- 全局侧边栏/抽屉状态。
- 非敏感显示偏好。

筛选条件放在 URL，表单临时值放在组件内部，不重复写入全局 Store。

## 11. API 协作规范

统一响应结构建议为：

```json
{
  "code": "OK",
  "message": "success",
  "data": {},
  "requestId": "01J..."
}
```

分页结构：

```json
{
  "items": [],
  "page": 1,
  "pageSize": 20,
  "total": 120,
  "totalPages": 6
}
```

Axios 拦截器负责：

- 添加请求 ID 或必要请求头。
- 携带同域 Cookie。
- 将网络错误、HTTP 错误和业务错误转换为统一 `AppError`。
- 401 清理登录状态并跳转登录页。
- 403 展示无权限页。
- 429 提示操作频繁。
- 5xx 展示通用提示并记录 `requestId`。

接口类型优先根据后端 OpenAPI 生成，生成文件禁止手工修改。

## 12. 登录与权限

推荐采用短期 Access Token + Refresh Token，均由后端通过安全 Cookie 管理：

- `HttpOnly` 防止 JavaScript 读取。
- 生产环境启用 `Secure`。
- 根据部署域名配置 `SameSite`。
- 状态变更接口配置 CSRF 防护或严格的同源策略。

应用启动时请求 `/api/v1/auth/me` 恢复登录状态。前端权限只控制体验，任何资源访问仍由后端校验。

## 13. 异常、加载与空状态

每个异步页面必须覆盖：

- 初次加载：骨架屏或 Spin。
- 局部刷新：保留旧数据并显示轻量状态。
- 空数据：给出与当前操作相关的下一步。
- 请求失败：显示可理解信息和重试入口。
- 无权限：独立 403 页面。
- 资源不存在：独立 404 页面。
- 服务异常：显示 `requestId` 便于排查。

提交按钮请求期间禁用，避免重复提交；不可重试的校验错误不进行自动重试。

## 14. 可访问性与响应式

- 表单控件必须有可识别标签。
- 图标按钮提供 `aria-label`。
- 弹窗打开后正确管理焦点。
- 键盘可以完成搜索、表单和主要操作。
- 文本与背景保持足够对比度。
- 商品图片提供替代文本。
- 支持 360px 宽度手机、常见平板和桌面浏览器。

4. 管理员下架商品 → C 端不可加购。

## 16. 构建与部署

两个应用分别构建静态资源，由 Nginx 提供访问和反向代理：

```text
mall-web image  → /api 转发到 commerce-api
admin-web image → /api 转发到 commerce-api
```

环境变量示例：

```text
VITE_API_BASE_URL=/api
VITE_AI_SEARCH_ENABLED=false
```

环境变量不得包含数据库密码和大模型密钥。Docker 构建采用多阶段构建，生产镜像只包含编译产物和 Nginx。

## 17. AI 扩展预留

第一阶段只做以下预留：

- 搜索条件定义为统一 `ProductSearchCondition`，不绑定 MySQL 实现。
- 商品 DTO 保留标签、使用场景、适用人群和规格参数。
- 商品卡片允许未来增加推荐理由区域，但本阶段不渲染。
- 使用 `VITE_AI_SEARCH_ENABLED` 控制未来入口。
- 搜索结果、收藏和购物车组件均可被 AI 结果页复用。

不创建空壳 AI 页面，不向用户展示无法使用的 AI 按钮。

## 18. 第一阶段验收标准

- C 端与管理后台可以独立构建和运行。
- 用户可以完成注册、登录、搜索、筛选、收藏和购物车操作。
- 管理员可以维护分类、品牌和商品，并控制商品上下架。
- 商品上下架、库存变化能正确反馈到 C 端。
- 页面与本地原型的核心布局、色彩和交互语言一致。
- 手机和桌面端核心功能可用。
- 核心流程 E2E 测试通过。
- 关闭 AI 功能开关后不存在不可用入口。
