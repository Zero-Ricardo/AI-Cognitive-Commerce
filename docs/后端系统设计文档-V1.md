# AI 商品采购平台第一阶段后端系统设计文档

> 文档版本：V1.0  
> 阶段范围：第一阶段——传统电商 MVP  
> 技术基线：Java 21 + Spring Boot + MySQL  
> 架构形态：模块化单体

## 1. 文档目的

本文档描述第一阶段后端的系统架构、模块职责、接口、数据模型、事务、安全、异常、测试和部署方案，作为后端实现、前后端联调和验收依据。

第一阶段不依赖 Redis、Elasticsearch、PostgreSQL 或 Python AI 服务。系统必须仅依靠 Spring Boot 和 MySQL 完成传统电商核心功能，并通过抽象接口为后续搜索增强和 AI 服务接入预留边界。

## 2. 建设原则

1. **先完成业务闭环**：商品、用户、搜索、收藏和购物车可独立运行。
2. **模块化单体优先**：避免第一阶段承担分布式事务、服务治理和多仓库协作成本。
3. **MySQL 是事实来源**：商品价格、库存、状态均以 MySQL 为准。
4. **接口与实现解耦**：搜索和热门商品通过接口抽象，后续可替换为 ES 和 Redis。
5. **安全默认开启**：所有管理接口和用户私有数据接口都必须后端鉴权。
6. **删除尽量可恢复**：商品等核心实体采用逻辑删除和状态控制。
7. **契约优先**：REST API、错误码和分页结构保持统一。

## 4. 系统上下文

```text
C端 React 应用 ─┐
                ├── HTTPS/JSON ── Spring Boot ── MySQL
后台 React 应用 ─┘
```

第一阶段 Spring Boot 同时提供 C 端和后台 API。后台接口使用 `/api/v1/admin/**` 命名空间，普通用户不能访问。

## 5. 工程结构

建议使用 Maven 多模块或单工程分包实现模块化单体：

```text
backend/
├── pom.xml
├── application/                 # 启动和模块装配
├── common/                      # 通用响应、异常、工具、审计
├── auth/                        # 注册、登录、令牌、权限
├── user/                        # 用户资料
├── catalog/                     # 商品、分类、品牌、图片、规格
├── search/                      # 商品搜索抽象及MySQL实现
├── cart/                        # 购物车
├── favorite/                    # 收藏
├── popularity/                  # 第一阶段热门商品统计
├── admin/                       # 后台用例入口
└── infrastructure/              # DB、文件存储、外部适配器
```

每个业务模块内部建议按职责分层：

```text
controller/       HTTP适配
application/      用例编排、事务边界
domain/           领域规则和接口
infrastructure/   Repository、Mapper和外部实现
dto/              请求与响应契约
```

模块只能通过公开 Application Service 或领域接口协作，禁止 Controller 直接操作 Mapper。

## 6. 业务模块设计

### 6.1 认证与用户

职责：

- 用户注册。
- 用户和管理员登录。
- Access Token 刷新。
- 退出登录和 Refresh Token 失效。
- 当前用户查询。
- 基础资料修改。
- 用户状态校验。

角色：

```text
USER   普通用户
ADMIN  管理员
```

第一阶段不设计复杂权限树，但数据库保留用户、角色及关联表，管理接口通过方法级权限校验。

### 6.2 商品目录

职责：

- 商品、分类、品牌、图片、规格和标签维护。
- 商品上下架。
- 商品详情查询。
- C 端只返回已上架且未删除商品。
- 管理后台可按状态查询商品。

商品状态：

```text
DRAFT       草稿
ON_SALE     已上架
OFF_SALE    已下架
```

删除采用 `deleted` 标识。已删除商品不可恢复到前台，管理操作需要审计。

### 6.3 商品搜索

定义稳定接口：

```java
public interface ProductSearchService {
    PageResult<ProductSummary> search(ProductSearchCondition condition);
}
```

第一阶段实现 `MysqlProductSearchService`，支持：

- 名称、商品编号、副标题和关键词的模糊匹配。
- 分类、品牌、价格区间和库存筛选。
- 综合、价格、上架时间和热度排序。
- 分页。

所有排序字段必须使用白名单映射，禁止直接把前端字符串拼接进 SQL。

### 6.4 收藏

职责：收藏、取消收藏、判断收藏状态和分页查看收藏。数据库唯一约束保证同一用户不能重复收藏同一商品。

商品下架后仍可在收藏列表显示历史条目，但必须标记为下架且禁止加入购物车。

### 6.5 购物车

职责：

- 添加商品。
- 修改数量。
- 删除条目。
- 单选、全选。
- 查询购物车及价格汇总。

规则：

- 同一用户同一商品只有一个购物车条目，再次添加时累加数量。
- 数量必须大于 0。
- 数量不能超过当前库存。
- 下架或删除商品不能加入购物车。
- 返回购物车时重新读取当前商品价格和状态，不把购物车中的历史价格当作当前价格。
- 第一阶段不锁定库存，不产生订单。

### 6.6 热门商品

第一阶段定义：

```java
public interface HotProductService {
    List<ProductSummary> getHotProducts(Integer categoryId, int limit);
}
```

MySQL 实现按照浏览、收藏和加购行为的加权分数计算：

```text
浏览 = 1 分
收藏 = 3 分
加入购物车 = 5 分
```

没有数据时按上架时间降级。第二阶段可以替换为 Redis Sorted Set，但 Controller 和前端接口保持不变。

## 7. API 设计

### 7.1 通用规范

- 基础路径：`/api/v1`。
- Content-Type：`application/json`。
- 时间：ISO 8601，服务端统一使用 UTC 存储和传输，前端按时区展示。
- 金额：接口使用十进制字符串或明确的两位小数，Java 使用 `BigDecimal`。
- ID：对外可使用 `Long`，前端 TypeScript 类型统一按字符串处理可规避大整数精度问题。
- 分页从 1 开始，`pageSize` 设置最大值，例如 100。
- 所有响应包含 `requestId`。

统一响应：

```json
{
  "code": "OK",
  "message": "success",
  "data": {},
  "requestId": "01J..."
}
```

## 8. 数据模型

### 8.1 用户与权限

#### `users`

| 字段 | 类型 | 说明 |
|---|---|---|
| id | BIGINT | 主键 |
| username | VARCHAR(64) | 用户名，唯一 |
| phone | VARCHAR(32) | 手机号，可空、唯一 |
| email | VARCHAR(128) | 邮箱，可空、唯一 |
| password_hash | VARCHAR(255) | 密码摘要 |
| nickname | VARCHAR(64) | 昵称 |
| status | VARCHAR(20) | ACTIVE/DISABLED |
| version | INT | 乐观锁 |
| created_at | DATETIME(3) | 创建时间 |
| updated_at | DATETIME(3) | 更新时间 |

#### `roles`、`user_roles`

保存 `USER`、`ADMIN` 角色及用户角色关联。管理员和普通用户可共用账户体系。

#### `refresh_tokens`

仅保存 Refresh Token 摘要、用户、过期时间和撤销状态，不保存可直接使用的明文 Token。

### 8.2 商品目录

#### `categories`

主要字段：`id`、`parent_id`、`name`、`level`、`sort_order`、`status`、`deleted`、时间戳。第一阶段限制最多两级。

#### `brands`

主要字段：`id`、`name`、`logo_url`、`description`、`sort_order`、`status`、`deleted`、时间戳。

#### `products`

| 字段 | 类型 | 说明 |
|---|---|---|
| id | BIGINT | 主键 |
| product_no | VARCHAR(64) | 商品编号，唯一 |
| name | VARCHAR(200) | 商品名称 |
| subtitle | VARCHAR(300) | 副标题 |
| category_id | BIGINT | 分类 |
| brand_id | BIGINT | 品牌，可空 |
| sale_price | DECIMAL(12,2) | 销售价 |
| original_price | DECIMAL(12,2) | 原价，可空 |
| stock | INT | 库存 |
| main_image_url | VARCHAR(512) | 主图 |
| description | TEXT | 商品描述 |
| keywords | VARCHAR(500) | 搜索关键词 |
| specification_json | JSON | 规格参数 |
| status | VARCHAR(20) | 商品状态 |
| published_at | DATETIME(3) | 上架时间 |
| version | INT | 乐观锁 |
| deleted | BOOLEAN | 逻辑删除 |
| created_at/updated_at | DATETIME(3) | 时间戳 |

#### `product_images`

保存商品图片地址、类型和排序值。

#### `tags`、`product_tags`

用于商品标签。使用场景和适用人群可以分别使用标签类型 `SCENARIO`、`AUDIENCE`，为后续 AI 检索准备结构化数据。

### 8.3 收藏和购物车

#### `favorites`

字段：`id`、`user_id`、`product_id`、`created_at`。建立 `(user_id, product_id)` 唯一索引。

#### `cart_items`

字段：`id`、`user_id`、`product_id`、`quantity`、`selected`、`version`、时间戳。建立 `(user_id, product_id)` 唯一索引。

### 8.4 行为记录

#### `product_events`

| 字段 | 说明 |
|---|---|
| id | 主键 |
| user_id | 登录用户，可空 |
| anonymous_id | 匿名标识，可空 |
| product_id | 商品 |
| event_type | VIEW/FAVORITE/ADD_CART |
| score | 对应分值 |
| occurred_at | 发生时间 |

第一阶段可用于热门商品统计。匿名标识只用于统计，不用于绕过隐私或跨站跟踪。

### 8.5 索引建议

- `products(status, deleted, category_id, brand_id)`。
- `products(sale_price)`。
- `products(published_at)`。
- `products(name)`；模糊包含查询不能完全依赖普通索引，数据增长后迁移 ES。
- `favorites(user_id, created_at)`。
- `cart_items(user_id, updated_at)`。
- `product_events(product_id, event_type, occurred_at)`。

具体组合索引必须基于实际 SQL 和 `EXPLAIN` 验证，避免无依据堆叠索引。

## 9. 核心流程与事务

### 9.1 注册

```text
校验请求
→ 标准化用户名/手机号/邮箱
→ 检查唯一性
→ BCrypt/Argon2id 哈希密码
→ 写入用户和默认角色
→ 返回注册结果
```

用户和角色关联在同一事务中提交。数据库唯一索引作为并发重复注册的最终保护。

### 9.2 新增或编辑商品

```text
管理员鉴权
→ 校验分类和品牌状态
→ 校验价格、库存、标签和图片
→ 保存商品及关联数据
→ 写管理审计日志
→ 提交事务
```

编辑请求携带 `version`。版本不一致返回 `RESOURCE_CONFLICT`，提示管理员刷新后重试。

### 9.3 上下架

上架前必须满足：

- 名称、分类、价格、库存和主图有效。
- 分类已启用。
- 关联品牌未被禁用。
- 商品未删除。

下架后，C 端搜索立即不返回该商品；已存在的收藏和购物车条目保留并显示不可购买状态。

### 9.4 加入购物车

```text
用户鉴权
→ 查询商品并校验状态
→ 校验目标数量和库存
→ 不存在则插入，存在则原子累加
→ 记录 ADD_CART 事件
→ 返回最新购物车摘要
```

并发情况下依赖唯一索引和事务重试防止重复条目。

### 9.5 收藏

收藏操作应具备幂等性：重复 `PUT` 仍返回成功，重复 `DELETE` 也返回成功。首次收藏时记录行为事件。

### 9.6 商品搜索

```text
规范化参数
→ 校验价格区间和分页
→ 排序字段白名单映射
→ 构造 MySQL 查询
→ 查询分页数据
→ 组装 ProductSummary
```

关键词最大长度、分页大小和筛选项数量需要限制，防止异常查询拖垮数据库。

## 10. 一致性和并发

- 商品编辑使用乐观锁。
- 购物车唯一约束保证单商品单条目。
- 库存第一阶段仅校验，不预占；订单阶段再设计库存锁定与扣减。
- 分类和品牌删除前检查子分类或商品引用。
- 价格使用 `BigDecimal`，数据库使用 `DECIMAL`，禁止使用 `double`。
- 跨模块多表写入由 Application Service 定义事务边界。
- 不在数据库事务内进行不可控的远程调用。

## 11. 安全设计

### 11.1 认证

- 密码使用 BCrypt 或 Argon2id 哈希，不可逆加密。
- Access Token 有效期较短。
- Refresh Token 轮换并保存摘要。
- Token 通过 `HttpOnly`、生产环境 `Secure` Cookie 下发。
- 退出登录撤销 Refresh Token。
- 被禁用用户不能登录或刷新令牌。

### 11.2 授权

- `/api/v1/admin/**` 仅允许 `ADMIN`。
- 收藏、购物车和个人资料只能访问当前用户数据。
- Repository 查询必须带当前用户 ID，避免仅凭资源 ID 操作造成越权。
- 文件上传接口校验管理员权限、文件类型、大小和真实内容。

### 11.3 输入和输出安全

- DTO 使用白名单字段，禁止 Entity 直接接收请求。
- 所有请求执行长度、格式、枚举和范围校验。
- 富文本如启用必须清洗 HTML；第一阶段优先使用纯文本或 Markdown。
- 日志不记录密码、Token 和完整敏感信息。
- 登录、注册、搜索和文件上传接口配置频率限制；第一阶段可使用应用内限流，第二阶段迁移 Redis。

## 12. 错误处理

使用全局异常处理器返回稳定错误码：

| 错误码 | HTTP | 场景 |
|---|---:|---|
| `VALIDATION_ERROR` | 400 | 参数不合法 |
| `UNAUTHORIZED` | 401 | 未登录或令牌失效 |
| `FORBIDDEN` | 403 | 权限不足 |
| `RESOURCE_NOT_FOUND` | 404 | 商品等资源不存在 |
| `RESOURCE_CONFLICT` | 409 | 唯一冲突或版本冲突 |
| `PRODUCT_OFF_SALE` | 409 | 商品已下架 |
| `INSUFFICIENT_STOCK` | 409 | 库存不足 |
| `TOO_MANY_REQUESTS` | 429 | 请求过频 |
| `INTERNAL_ERROR` | 500 | 未预期错误 |

生产环境不得把 SQL、堆栈、类名或数据库约束名返回前端。日志通过 `requestId` 与响应关联。

## 13. 日志、审计与监控

### 13.1 应用日志

采用结构化日志，至少包含：

- 时间。
- 日志级别。
- `requestId`。
- 用户 ID，可用时。
- HTTP 方法和路径。
- 状态码和耗时。
- 业务错误码。

### 13.2 管理审计

建立 `admin_audit_logs`，记录管理员、操作类型、资源类型、资源 ID、变更摘要、请求 ID 和时间。密码及敏感字段不进入变更内容。

### 13.3 健康检查

Spring Boot Actuator 提供：

- 存活检查。
- 就绪检查。
- MySQL 连接状态。
- 基础指标。

管理端点不能全部公开到公网。

## 14. 文件与图片存储

业务表只保存文件标识和访问 URL，不将图片二进制写入 MySQL。

定义存储接口：

```java
public interface FileStorageService {
    StoredFile store(ImageUpload upload);
    void delete(String fileKey);
}
```

第一阶段可以使用本地挂载目录实现，Docker 通过 Volume 持久化；部署到生产前可替换为 MinIO 或云对象存储。需要限制 MIME 类型、文件大小和文件名，禁止执行上传内容。

## 15. 数据迁移与初始化

- 所有 DDL 由 Flyway 管理。
- 禁止应用启动时使用 ORM 自动修改生产表结构。
- 初始化脚本创建基础角色、管理员和演示分类。
- 管理员初始密码通过环境变量或一次性初始化流程传入，不写死在代码和 SQL 中。
- 开发演示数据与正式迁移脚本分离。

## 17. 性能目标

在约定的单机验收环境和合理测试数据量下：

- 普通商品列表 P95 小于 1 秒。
- 商品详情 P95 小于 500 毫秒。
- 购物车查询 P95 小于 500 毫秒。
- 登录接口 P95 小于 1 秒，不含异常网络延迟。
- 分页大小默认 20，最大 100。
- 单次关键词长度和复杂筛选数量有限制。

目标需在确定硬件、并发数和数据规模后转换为正式压测指标。

## 19. 后续增量扩展

### 19.1 Elasticsearch

Redis 数据必须可以从 MySQL 行为记录重建。

### 19.3 Python AI 服务

后续 AI 阶段由 Spring Boot 统一接入 Python AI 服务，第二阶段暂不实现：

```text
自然语言 → AI结构化意图 → 现有ProductSearchService → 商品结果
```

AI 不直接写商品、库存、购物车和收藏。第一阶段的标签、场景、适用人群和规格字段将作为 AI 搜索的数据基础。
